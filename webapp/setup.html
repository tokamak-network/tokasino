실행 레이어(EL)와 프리컴파일
실행 레이어(EL)는 스마트 컨트랙트를 실행하는 엔진입니다. 이더리움의 Geth 같은 역할인데, 우리는 op-reth를 포크해서 커스텀했어요.

프리컴파일(Precompile)이란?
솔리디티로 짜면 느리거나 불가능한 연산을 EVM 엔진 내부에 네이티브 코드로 미리 구현해둔 것입니다. 특정 주소를 호출하면 바로 실행돼요. 이더리움에 이미 있는 예시:
  0x01 = ecrecover (서명 복원)
  0x02 = SHA-256 해시
  ... 0x0a까지 사용 중

왜 0x0b인가요?
기존 프리컴파일이 0x01~0x0a까지 사용 중이라, 충돌 없이 바로 다음 번호인 0x0b(11번)에 우리의 난수 생성기를 배치했습니다.

seed는 어디서 오나요? (가장 중요!)
프리컴파일이 받는 seed는 block.prevrandao에서 옵니다. 이것이 Enshrined VRF 난수의 신뢰의 근원이에요.

prevrandao란?
모든 이더리움 블록에는 prevrandao라는 필드가 있습니다. 원래 이더리움에서는 비콘체인의 RANDAO 믹스 값이 여기에 들어가요. 하지만 Enshrined VRF에서는 이 자리에 시퀀서가 VRF로 계산한 난수를 넣습니다.

VRF 난수가 만들어지는 과정:
  이전 블록 해시(0xab12...) + 블록번호(N)
    ↓ 시퀀서의 BLS 비밀키로 서명
  96바이트 BLS 서명(=증명)
    ↓ keccak256
  0x3a8f... = prevrandao (32바이트)
    ↓ 이 값이 블록에 기록됨
    ↓ 솔리디티에서 block.prevrandao로 읽음
    ↓ 프리컴파일의 seed로 사용!

왜 이것이 안전한가요?
• 결정적: 같은 입력이면 항상 같은 결과. 시퀀서가 마음대로 바꿀 수 없음
• 예측 불가: 비밀키 없이는 다음 블록의 값을 알 수 없음
• 검증 가능: 공개키로 누구나 "이 난수가 진짜인지" 확인 가능
• 체이닝: 블록 N의 해시 → 블록 N+1의 VRF 입력 → 연쇄적으로 연결

즉, 시퀀서조차 난수를 조작할 수 없는 구조입니다. VRF는 "서명"이기 때문에 입력이 정해지면 출력이 하나로 결정돼요. 다른 값을 내놓으면 공개키로 즉시 들통납니다.

그런데 왜 프리컴파일에서 한번 더 가공하나요?
prevrandao 자체는 블록당 하나의 고정값이에요. 같은 블록의 모든 트랜잭션이 같은 prevrandao를 봅니다. 그래서 프리컴파일에서 카운터를 섞어서 호출마다 다른 난수를 만듭니다:
  1번째 호출: keccak256(prevrandao + 0) → 난수 A
  2번째 호출: keccak256(prevrandao + 1) → 난수 B
  3번째 호출: keccak256(prevrandao + 2) → 난수 C
 카운터 없으면? 한 블록에서 주사위 3개 굴리면 3개 다 같은 숫자!

왜 ChaCha20을 거치나요?
3. ChaCha20 난수 생성기
Step 2의 해시를 ChaCha20이라는 CSPRNG의 시드(씨앗)로 넣습니다.

CSPRNG = 암호학적으로 안전한 의사 난수 생성기
(Google이 TLS/HTTPS에 사용하는 것과 같은 알고리즘)

왜 keccak256 해시를 그대로 안 쓰고 한 단계 더 거칠까요?
• keccak256은 입력을 알면 출력을 계산할 수 있습니다 (해시니까요).
• ChaCha20을 거치면 입력과 출력 사이의 관계가 완전히 끊어집니다.
• 즉 이중 자물쇠. seed를 알아도 최종 난수 예측이 불가능!

4. 32바이트 난수 반환
ChaCha20이 32바이트(256비트) 난수를 솔리디티에 반환합니다.

  전체 흐름:
  seed + counter
    ↓ keccak256
  0x91f3a7... (해시)
    ↓ ChaCha20 시드
  0x7d2e3f... (최종 난수!)
    가스: 100 (네이티브라 초고속)

VRF가 뭔가요?
VRF(Verifiable Random Function)는 검증 가능한 난수 함수입니다. 비밀키로 서명하면 랜덤한 값이 나오는데, 누구나 공개키로 "이 값이 정말 그 키로 만들어졌는지" 확인할 수 있어요.

핵심: 같은 입력 = 항상 같은 출력 (조작 불가), 하지만 비밀키 없이는 출력을 예측할 수 없음. 카지노에 딱 맞는 성질이죠!

Engine API가 뭔가요?
합의 레이어(CL)가 실행 레이어(EL)에게 "이 블록을 만들어"라고 지시하는 통로입니다. 이더리움 2.0에서 비콘체인이 실행 클라이언트에 연결하는 것과 같은 방식이에요.

여기서 중요한 건: CL이 VRF로 만든 랜덤 값을 prevRandao 필드에 넣어서 EL에 전달한다는 것. 즉 블록의 난수는 합의 레이어가 결정합니다.

왜 제네시스 해시가 필요한가요?
VRF는 매 블록 이전 블록의 해시 + 블록 번호를 입력으로 받아요. 그런데 첫 번째 블록은 "이전 블록"이 없잖아요? 그래서 제네시스(0번) 블록의 해시를 시작점으로 사용합니다.

이렇게 하면 블록 1의 난수 → 블록 2의 입력 → 블록 2의 난수 → ... 로 체인처럼 연결되어, 어떤 블록의 난수도 조작할 수 없게 됩니다.

First VRF Block

첫 블록은 어떻게 만들어지나요?
시퀀서가 VRF(genesis_hash || 1)을 계산합니다.

1. 입력값 준비: 제네시스 해시 + 블록번호 1
2. BLS 비밀키로 서명 → 96바이트 증명(proof) 생성
3. keccak256(증명) → 32바이트 난수 출력
4. 이 값을 prevrandao로 넣어 블록 생성
5. Engine API로 EL에 전달 → 블록 #1 확정!

이후부터는 같은 과정이 매 블록(3초)마다 반복됩니다.

컨트랙트는 어떻게 난수를 쓰나요?
솔리디티에서 block.prevrandao를 읽으면 시퀀서가 VRF로 만든 난수를 바로 가져올 수 있어요.

각 게임 컨트랙트는 이 값에 플레이어 주소, 게임 번호 등을 섞어서 게임마다 고유한 난수를 만듭니다. 하우스에 10 ETH씩 넣어두면 당첨금을 바로 지급할 수 있어요.
